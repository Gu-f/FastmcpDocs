FastMCP 会频繁发布，以便在快速演进的 MCP 生态中尽快交付功能。我们以务实方式使用语义化版本：Model Context Protocol 仍然年轻，模式仍在形成，等待完美稳定意味着会错过用更好工具赋能开发者的机会。

## 版本策略

### 语义化版本

**Major（x.0.0）**：完整 API 重新设计

Major 版本代表根本性变化。FastMCP 2.x 在实现和设计理念上都与 1.x 完全不同。

**Minor（2.x.0）**：新功能和演进

<Warning>
不同于传统语义化版本，minor 版本在生态演进需要时**可能**包含[破坏性变更](#破坏性变更)。在年轻生态中，这种灵活性很重要，因为完美向后兼容会阻碍重要改进。
</Warning>

FastMCP 始终面向最新的 MCP Protocol 版本。MCP 规范或 MCP SDK 中的破坏性变更会自动传导到 FastMCP；相比维护旧协议版本兼容性，我们更优先保持对最新功能和约定的跟进。

**Patch（2.0.x）**：bug 修复和细节改进

Patch 版本只包含 bug 修复，不包含破坏性变更。你可以放心应用这些安全更新。

### 破坏性变更

我们允许 minor 版本包含破坏性变更，因为 MCP 生态正在快速演进。拒绝打破有问题的 API 会累积设计债，最终让框架变得不可用。每个破坏性变更都代表一个有意决策：让 FastMCP 与生态演进保持一致。

发生破坏性变更时：
- 它们只会出现在 minor 版本中（例如从 2.3.x 到 2.4.0）
- 发布说明会解释变更内容和迁移方式
- 在可行时，我们会至少提前 1 个 minor 版本提供弃用警告
- 变更必须能为用户带来实质收益，才足以证明这种扰动是合理的

公共 API 是我们兼容性保证覆盖的范围，也就是你可以依赖其在同一个 minor 版本内保持稳定的 FastMCP 部分。公共 API 包括：
- `FastMCP` 服务端类、`Client` 类和 FastMCP `Context`
- 核心 MCP 组件：`Tool`、`Prompt`、`Resource`、`ResourceTemplate` 和 transports
- 它们的公共方法和已记录行为

其他所有内容（实用工具、私有方法、内部模块）都可能不经通知而变化。这个边界让我们可以重构内部实现并改进细节，而不破坏你的代码。为了生产稳定性，请固定到具体版本。

<Warning>
`fastmcp.server.auth` 模块在 2.12.0 中引入，暂时豁免于此策略，这意味着即使在 patch 版本中也*预期*可能出现破坏性变更。这是因为 auth 是 MCP 规范中快速演进的一部分，受限于旧决策会带来风险。如果你在生产中使用身份认证，请固定 FastMCP 版本。

我们预计该豁免至少会持续到 2.12.x 和 2.13.x 发布系列结束。
</Warning>

### 生产使用

固定到精确版本：
```
fastmcp==2.11.0  # 好
fastmcp>=2.11.0  # 不好：会安装破坏性变更
```

## 创建发布

我们的发布流程有意保持简单：

1. 使用 `vMAJOR.MINOR.PATCH` 标签创建 GitHub release（例如 `v2.11.0`）
2. 自动生成发布说明，并按需整理或添加额外编辑信息
3. GitHub release 会自动触发 PyPI 部署

这种自动化让维护者能够专注于代码质量，而不是发布机制。

### 发布节奏

我们遵循功能驱动的发布节奏，而不是固定日程。当重要功能准备就绪时，minor 版本大约每 3-4 周发布一次。

Patch 版本会针对以下情况及时发布：
- 关键 bug 修复
- 安全更新（立即发布）
- 回归修复

这种方式意味着改进一旦准备好你就能获得，而不是等待任意设定的发布日期。
